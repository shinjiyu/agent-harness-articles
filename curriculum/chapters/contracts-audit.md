# 6. 从提示到合约：可审计治理

企业落地时，提示 + 检索原型会撞上：源边界、实体路由、应答合同、可复现跟踪。Harness 工程把确定性行为下沉到 **代码、清单、schema、验证工件**，让有源声明仍是答案权威。

## 模式要点

| 工件 | 作用 |
|------|------|
| Schema / 清单 | 组合边界上的可替换合约 |
| 验证器 | 故障注入与回归门禁 |
| 跟踪 | 可审计轨迹，而非聊天日志 |
| 过程逻辑（如 LTL） | 程序本身是正确性的一部分 |
| 事务式准入 | 生成操作在约束下才提交 |

关键消融结论（来自合约向论文的典型主张）：**纯提示往往复制不了代码层保证**；外挂护栏可能过度拒绝，损伤效用。

### 关键要点

- 可审计性与效率常落在同一组件（编排 / 控制面）
- 合约要可版本化、可对模型替换回归
- 下方论文：Prompts→Contracts、AgentLTL、Mnemosyne、形式验证等
