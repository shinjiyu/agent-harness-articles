# 4. 上下文工程与记忆

上下文工程回答一个问题：**每次推理让模型看见什么？**

Harness 负责：组装、压缩、卸载、恢复、以及跨回合的记忆合约。窗口越大不等于越好——嘈杂长上下文会同时抬成本与伤质量（lost-in-the-middle / context rot）。

## 常见机制族

| 机制 | 作用 |
|------|------|
| 前缀缓存友好拼装 | 稳定部分可反复命中 cache |
| 增量 compaction | 旧历史 → 可恢复摘要 + 近尾原文 |
| 工具输出卸载 | 文件系统存全文，上下文只留指针 |
| 有界记忆合约 | 按类型检索组装，而不是无限拼接 transcript |
| 主动记忆代理 | 在关键时刻注入提醒，对抗行为状态衰减 |
| Self-GC / 对象生命周期 | 把上下文当可回收对象，而不只是文本后缀 |

## 设计原则

1. **决策相关优先**：进窗口的应能改变下一步动作
2. **可恢复**：压缩不能变成不可审计的黑箱摘要
3. **父子隔离**：子代理探索不要污染父环上下文
4. **显式合约**：每层记忆允许看见什么，写进规格

### 关键要点

- 上下文策略同时影响成本与正确性
- 记忆不是「更大窗口」，而是生命周期与注入策略
- 对照下方论文：GRACE、主动记忆、Self-GC、AgenticSTS 等
