# 7. 评估、基准与 Harness 偏差

软件代理基准常只报告「是否完成任务」。但代理通过 harness 到达结果——看见什么、能做什么、修什么、验什么、记什么，都会改写多步信念。

## 必须建立的意识

1. **Harness 是实验变量**，不是可忽略实现细节  
2. **工具调用排行榜分数 ≠ 能力**：评测器本身可能有 10%+ 与人评分歧，以及巨大的运行间方差  
3. 应分解指标：工具调用 / 任务完成 / 结果验证  
4. 发布门禁建议同时看质量与 **CPM / 成本**

## 实践清单

- 锁任务、锁模型、只换 harness，才能归因
- 重复运行报告方差，尤其是 LLM-as-judge
- 对「规格不足」场景单独测动作边界违规（猜动作）
- 小样本质量 Δ 只能当方向性，不要写成显著性

### 关键要点

- 先怀疑评测器，再相信排行榜
- 信念分歧诊断帮助解释「成功但决策已偏」
- 下方论文：Belief Divergence、Benchmarking the Benchmarks、UnderSpec 等
