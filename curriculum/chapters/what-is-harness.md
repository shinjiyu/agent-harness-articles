# 1. 什么是 Agent Harness

Harness Engineer 关注的不是「Agent 会不会说话」，而是「Agent 在生产环境能不能稳定干活」——环境、约束、工具、记忆、评估与反馈闭环。

## 和常见角色的差别

| 角色 | 关注点 |
|------|--------|
| 算法 / 研究 | 模型能力上限 |
| 产品 | 用例与体验 |
| DevOps | 部署与可用性 |
| **Harness / Agent Engineer** | 让模型在真实约束下可靠完成任务 |

## 核心能力面

| 能力面 | 一句话 |
|--------|--------|
| 上下文工程 | 决定每次推理装进窗口的东西 |
| 约束设计 | 边界、审批门、失败策略 |
| 工具编排 | 暴露什么工具、何时调用、如何验证 |
| 规格治理 | 把规格当活合约版本化 |
| 质量闭环 | 观测 → 评估 → 修复 → 再评估 |

## 为什么 2026 更强调 Harness

单靠「更好的 prompt」撑不住 30～40 步轨迹、工具失败、长上下文衰减与成本爆炸。可靠性问题变成**系统问题**：要从编排层改写账单与行为，而不是只换更大底座。

### 关键要点

- Harness = 模型周围的运行支架，不是附属脚本
- 能力面覆盖上下文、约束、工具、规格、评估闭环
- 本站后续章节会把论文按这些面归类，方便「概念 ↔ 证据」对照
