# TTHE: Test-Time Harness Evolution

- ArXiv: [2607.08124](http://arxiv.org/abs/2607.08124)
- Tier: **A**
- Indexed: 2026-07-13T17:32:31+08:00

## 核心介绍

LLM 代理行为不仅取决于模型，还取决于 harness：构建上下文、调用工具、验证中间结果、从故障恢复的可执行程序。现有方法多在部署前优化 harness，再在测试时冻结；一旦测试分布或故障模式偏移，适应能力受限。

TTHE（Test-Time Harness Evolution）问：能否在评估期间，**仅用测试输入上产生的未标注执行轨迹**来优化 harness？

### 方法要点

- 把可执行 harness 当作测试时适应的**状态**
- 维护候选 harness 集合；提议者对执行轨迹推理（无金标、无任务特定监督）
- 裁判基于执行衍生的代理信号提交改进；选定程序持续管理后续输入
- **不更新模型权重**、不训练单独适应模型；求解器/提议者/裁判是同一冻结 LLM 上的不同角色与工具

### 覆盖任务（摘要）

文本转 SQL、竞赛编程、软件工程、数据科学编码、代理工具使用等；相对固定 ReAct 风格基线 harness 有持续、可检查的改进。

### 为何进 A 档

问题设定干净、约束明确（无权重更新、无金标），把测试时适应从「换模型/重试」改写为「演化可执行控制程序」，与本主题高度对齐。

### 阅读时注意

需关注跨任务稳定性、算力预算，以及与「预搜索工作流 / 按查询重试」的边界是否在全文中严格划清。
