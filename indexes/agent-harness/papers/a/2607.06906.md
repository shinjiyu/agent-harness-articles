# The Harness Effect: How Orchestration Design Sets the Token Economics of Enterprise Agentic AI

- ArXiv: [2607.06906](http://arxiv.org/abs/2607.06906) · [HTML](https://arxiv.org/html/2607.06906)
- 作者：Writer, Inc.（Muayad Sayed Ali, Waseem AlShikh 等）
- Tier: **A**
- Indexed: 2026-07-13T17:32:31+08:00
- 技术详解更新：2026-07-13

## 一句话

在任务与模型都锁死时，只换编排层（conventional production loop → Writer Agent Harness），可使混合成本约 −41%、token 约 −38%、时延约 −44%，质量大致持平；**效率增益几乎与模型无关，质量增益随模型能力线性上升（harness leverage）**。

---

## 1. 问题：Token Maxing

Agent 任务不是一次模型调用，而是多回合循环：system prompt、工具 schema、检索载荷、推理、工具输出，以及 naive 实现里对历史的全量重放。账单由**环绕模型的软件（harness）**决定，而不是模型本身。

行业默认对策是「用更多 token 买能力」：更长推理、更多回合、更大工具载荷、更大重放上下文。单价下跌掩盖了这一模式，但总支出仍上升——作者称之为 **token maxing**：

> token 强度 \(\tau\) 上升，同时边际质量/token 下降（比系统历史平均兑换率更差）。

多数已有效率工作打在**单次调用内**（压缩、预算推理、投机解码）或**模型间路由**；二者都把编排层当作给定。本文主张：对抗 token maxing 的决定性杠杆是 harness。

## 2. 形式化：账单拆到编排层

### 2.1 任务成本

对 \(k\) 回合循环，回合 \(i\) 输入/输出 token 为 \(T^{\mathrm{in}}_i, T^{\mathrm{out}}_i\)：

\[
C = \sum_{i=1}^{k}\big(p_{\mathrm{in}} T^{\mathrm{in}}_i + p_{\mathrm{out}} T^{\mathrm{out}}_i\big)
\]

### 2.2 输入由 harness 组装

\[
T^{\mathrm{in}}_i = S_i + H_i + G_i + R_i + U_i
\]

| 项 | 含义 | 谁控制 |
|----|------|--------|
| \(S\) | system | harness |
| \(H\) | history | harness |
| \(G\) | tool schemas | harness |
| \(R\) | retrieval | harness |
| \(U\) | user turn | 应用 |

Naive 全历史重放使累计输入近似 **\(O(k^2)\)**；compact / cache / offload 后可降到近似 **\(O(k)\)**。

### 2.3 Prompt cache 下的有效输入价

Agent 负载通常 **输入主导**（生产中 input:output 可近 100:1）。若 cache hit 比例为 \(h\)、缓存价约为 list 的 \(\kappa\approx 0.1\)：

\[
p^{\mathrm{eff}}_{\mathrm{in}} = p_{\mathrm{in}}\big(1 - h(1-\kappa)\big)
\]

\(h\) **不是模型属性**，而是 prompt 跨回合字节稳定性——完全由编排如何拼上下文决定。因此 harness 同时控制：**提交多少 token** 与 **主导部分按什么价计费**。

### 2.4 效率指标

\[
\eta_{\$}=Q/C,\qquad \mathrm{CPM}=Q\cdot 10^{6}/\tau
\]

本文建议把 **CPM（每百万 token 完成任务数）** 与质量一起作为 release gate，类比芯片的 performance-per-watt。

## 3. Writer Agent Harness 是什么

位于应用与任意基础模型之间的 runtime，负责：

1. **上下文组装**：system、会话状态、文档/知识图谱 grounding
2. **工具层**：原生工具 + MCP 连接器；schema 暴露与调用中介
3. **工作流**：多步 Playbooks 端到端执行
4. **委派**：生成 scoped sub-agent 并合并结果（baseline 没有）
5. **可观测性**：每回合记录 prompt/completion tokens、工具事件、墙钟时间；成本用**钉死的公开价表**离线算，两臂计价一致

模型是配置项 `model_name`，因此才能做「六模型 × 两编排 × 同一任务集」的受控交换。

### Baseline（2026-06-07 冻结的生产回路）对比

| 维度 | Baseline（行业默认） | Writer Harness |
|------|----------------------|----------------|
| System prompt | ~49KB，每回合全量重放 | 稳定前缀 + 易变尾部（两区 prompt） |
| 工具调用 | 文本流里 XML + 正则解析 | **仅原生 tool calling**（XML 路径删除） |
| 上下文溢出 | 破坏性 middle-truncation | 非破坏性结构化 compaction |
| 等待 | polling（持续烧 token） | 耐久挂起，零 token 等待 |
| 模型适配 | 按模型调 prompt | 单一执行路径，模型无关 |
| Sub-agents | 无 | 有（scoped + 合并） |

## 4. 六族机制：如何改写账单

设计目标一句话：**尽量让 token 满足 (a) 可缓存、(b) 决策相关、(c) 花在可提交、可恢复的工作里**——用结构强制，而不是指望模型自觉。

### (1) Cache-shape discipline：两区 Prompt

```
[ 字节稳定前缀 ]                    [ 每回合重建的易变尾 ]
 tool-schema 目录
 稳定 system prompt
 append-only 耐久 transcript        时钟 / 文件列表 / plan 状态
                                   一次性提醒 / voice / 自定义指令
```

- 易变内容**结构上禁止**进入前缀；cache breakpoint 不得落在首个易变消息之后
- 最多 4 个 provider breakpoint（schema 后、system 后、沿最近两段耐久回合滑动）
- 会话内约 1 小时 retention latch，避免中途策略翻转
- 文中实测：同前缀调用 7,876 / 7,886（**99.9%**）为 cache read → 主导输入项约 **0.1× list**

输入主导负载下，**cache hit rate 是 harness 可控的最高杠杆成本变量**。

### (2) 结构化、增量、cache-aware compaction

- 触发：约 **80%** 模型输入预算
- 折叠为 typed checkpoint：耐久记忆（决策/约束/已否方案）、八段可恢复执行摘要、 verbatim 用户需求、skill 引用
- 保留最近 **4–12** 条消息（≤30% 预算）原文
- 摘要跑在**更便宜的 helper 模型**上，离开付费主循环；空/劣化摘要则**中止** compaction
- 仍塞不下则确定性阶梯：先缩尾，再 middle-truncate +「重述要紧内容」提示
- 与 cache **共设计**：每回合重写历史会毁掉前缀稳定性；checkpoint 是耐久行，重建后的 prompt 成为新可缓存前缀

这是把 \(O(k^2)\) 历史重放压向近似 \(O(k)\) 的核心。

### (3) Context offload：模型不必付钱的信息

| 手段 | 合约要点 |
|------|----------|
| Sub-agents 作上下文防火墙 | 子代理在独立上下文检索；回传摘要 **≤8KB**；引用走 metadata sidecar，父模型不读全文 |
| Skills 渐进披露 | Prompt 只带 name+description 表；完整 skill 文档按需安装读取（inline seed ≤20K chars） |
| 大工具输出落盘 | Shell >20K：head+tail 预览 + 完整写文件；web fetch inline 8K；超大文件读拒绝而非静默截断 |
| Plan/canvas | 事件溯源，每回合投影一次紧凑渲染；工具结果用一行 ACK，避免状态在 transcript 重复 |
| 后台结果 | 恰好一次作为 ephemeral 尾提醒投递 |
| 媒体 | ≤4 张图或 2MB；更旧的驱逐并留 reload stub |

文件系统是无界记忆；上下文只持指针。

### (4) Zero-token waiting + 耐久性

- 等人、等审批、等长后台任务：**耐久挂起，0 token**；ingress 事件恢复，无 polling
- WAL：事件落盘后再流式发出；崩溃用 generation fencing 从下一序号恢复
- 工具结果先持久化再展示——避免「丢 40 回合等于重买 40 回合 token」

### (5) Failure-spend governance

- 失败先分类（rate limit / stall / timeout / 畸形流 / 宕机 / permanent），仅白名单类才 fallback 下一 provider
- 流式中途失败 = **discarded attempt**：清草稿，禁止副作用（工具执行、耐久写）
- Circuit breaker：同一字节级失败工具调用连续 3 次则停，并按原因引导（改参 vs 退避）
- 截断输出与长度上限以**显式 typed state** 终止，禁止静默
- 循环上限 **50**；工具并行上限 **4**

Retries / 死胡同 / doom loop 是公式 (1) 上的乘数；harness 限制乘数。

### (6) Model-agnostic floor

- 路由计划是**数据**（模型、provider、fallback 顺序）；循环从不按模型名分支
- 所有 provider 流归一成同一 chunk 合约
- 仅原生 tool calling；对弱模型做 schema hygiene：`$refs` 内联、双重编码 JSON 恢复、校验错误清洗、过载 schema 拆分

解释了效率增益的**模型不变性**：floor 由 harness 固定，ceiling 由模型能力决定。

## 6. 关键结果（技术解读）

### 6.1 混合效率（六模型平均）

| 指标 | Baseline | Harness | \(\Delta\) |
|------|----------|---------|------------|
| Quality | 0.78 | 0.81 | +0.03（视为持平） |
| Cost / task | $0.21 | $0.12 | **−41%** |
| Median wall-clock | 48 s | 27 s | **−44%**（约 1.8×） |
| Tokens / task | 14.2k | 8.8k | **−38%** |
| \(\eta_{\$}\) | 3.71 | 6.75 | **+82%** |
| CPM | 54.9 | 92.0 | **+68%** |

### 6.2 模型不变的成本下降

| 模型 | Cost \(\Delta\) | Latency \(\Delta\) |
|------|-----------------|-------------------|
| Claude Sonnet 4.6 | −39% | −41% |
| Gemini 3.1 | −33% | −40% |
| Gemini Flash 3.5 | **−61%** | **−55%** |
| Qwen 3.6 | −44% | −33% |
| GLM 5.1 | −47% | −38% |
| Palmyra X6 | −52% | −48% |

- **无一例外**：每模型至少便宜约 1/3 → 层效应签名
- 相对「换模型」：baseline 下最贵→最便宜约省 36%；**保留任意模型换 harness 省 33–61%**
- 快档模型相对收益最大：harness 开销（重放历史、广播 schema）占其账单比例更高

### 6.3 Harness Leverage（质量侧）

48 个能力×模型格子：30 提升、11 持平、7 回退。全部回退落在三个较弱模型，且集中在编排最重的能力（MCP / Playbooks / Presentations）。

各模型平均质量增益 vs baseline 强度：\(r=0.99\)（\(n=6\)，模式强但样本少）：

| 模型 | \(\Delta\bar{q}\) |
|------|-------------------|
| Palmyra X6 | +0.079 |
| Sonnet 4.6 | +0.073 |
| Gemini 3.1 | +0.050 |
| GLM 5.1 | +0.028 |
| Flash 3.5 | +0.010 |
| Qwen 3.6 | **−0.031** |

含义：**效率无条件；质量要能力去「赚」**。弱模型仍拿成本削减，但不一定变更好。

### 6.4 新能力的能力地板：Sub-agents

| Sonnet | Gemini 3.1 | Flash | Qwen | GLM | Palmyra |
|--------|------------|-------|------|-----|---------|
| 0.85 | 0.70 | 0.45 | 0.42 | 0.58 | 0.86 |

仅两强模型过「可用可靠性」门槛。编排功能带 **capability floor**：低于地板时暴露该功能会产生失败而非能力。Token 侧：子代理上下文 scoped，父环只付封顶摘要（防火墙合约）。

### 6.5 代表性任务纹理

| 任务 | 质量 | 备注 |
|------|------|------|
| Medicare 三回合 grounding | 0.60→0.90 | 最大质量跃迁；检索 shaping「更少但更相关」 |
| Identity/refusal | 0.90→0.90 | 质量不变，成本减半（$0.04→$0.02） |
| Contract Q&A | 0.75→0.82 | 提升 |
| Multi-step research | 0.80→**0.60** | 成本 −46%，但质量回退——聚合持平掩盖的真实权衡 |

## 7. 工程含义（可落地）

1. **账单是 \(p\times\tau\)**：比价只看 \$/Mtok 是在比 \(p\)；\(\tau\) 属于 harness。
2. **Harness 是可复利资产**：模型无关、随任务量线性、可与路由/压缩/降价叠加；文中粗算 100 万任务/月 ≈ 省 \$90k/月。
3. **按能力降级暴露功能**：弱模型缩小 tool catalog、关闭 delegation；路由应按「将行使的编排特征」而不只按题面难度。
4. **Release KPI**：效率（均匀、决定性）应作主标题；不要用 +0.03 质量做宣传口径。
5. **与其它框架对比视角**（文档级，非实测）：多数系统把 cache/compaction/offload/失败治理留给应用；无 per-task accounting 则 token maxing **不可观测 = 不可管理**。

## 8. 局限（阅读时必须打折）

- \(n=22\)：质量仅方向性；效率符号一致才「硬」
- Baseline 单次运行后冻结，未测 run-to-run 方差
- LLM-as-judge；价表钉死一份公开表（token/时延结论不依赖价）
- 任务偏企业助手（grounding/workflow/tools），未必外推到长程 coding bench
- harness leverage \(n=6\)；只比较**一对** baseline vs Writer harness（厂商利益披露见原文）
- 与其它系统的 Table 1 是设计对比，不是跨 harness 实测
